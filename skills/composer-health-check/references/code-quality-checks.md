# 代码质量检查参考手册

每条检查项都包含：查什么、为什么重要、怎么高效检测。

## 检查分类

| 优先级 | 类别 | 扫描策略 |
|--------|------|----------|
| 🔴 CRITICAL | 安全与类型 | grep 为主，速度快 |
| 🟡 HIGH | 现代化改造 | 模式匹配，针对性扫描 |
| 🔵 MEDIUM | 结构与性能 | 抽样检查，代表性文件 |

---

## 🔴 CRITICAL：安全与类型

### CQ-001：缺少 `declare(strict_types=1)`

**查什么**：PHP 文件开头没有 `declare(strict_types=1)`。

**为什么重要**：没有 strict types，PHP 会静默进行类型强制转换——`"123abc"` 变成 `123`，float 截断成 int。这类 bug 在 code review 中很难发现，生产环境出问题更难排查。

**怎么检测**：
```bash
grep -rL "declare(strict_types=1)" src/ --include="*.php"
```
`src/` 下的每个 `.php` 文件都应该有。例外：纯模板/HTML 文件（在库项目中几乎不存在）。

**怎么修**：在文件最开头（namespace 之前）加上 `declare(strict_types=1);`。

---

### CQ-002：缺少返回类型

**查什么**：方法/函数没有声明返回类型。

**为什么重要**：返回类型是自文档化的，也防止意外返回错误类型。PHP 8.0+ 支持 `void`、`never`、`static`、联合类型、nullable 类型。

**怎么检测**：
```bash
grep -rP "function\s+\w+\s*\(" src/ --include="*.php" | grep -v "):"
```
先看 public/protected 方法——它们定义了 API 契约。

**怎么修**：加上合适的返回类型。无返回值用 `void`，总是抛异常/exit 的方法用 `never`。

---

### CQ-003：缺少参数类型

**查什么**：方法/函数参数没有类型声明。

**为什么重要**：无类型参数什么都能接收，导致运行时报错发生在调用链深处，很难排查。

**怎么检测**：
```bash
grep -rP "function\s+\w+\s*\([^)]*\$" src/ --include="*.php"
```
检查没有类型声明的参数。如果类型确实多变，用 `mixed` 显式声明，比不写强。

**怎么修**：加上最具体的类型。能用联合类型（`string|int`）就不用 `mixed`。

---

### CQ-004：缺少属性类型

**查什么**：类属性没有类型声明。

**为什么重要**：无类型属性可以存任何值，静态分析和 IDE 自动补全全部失效。

**怎么检测**：
```bash
grep -rP "(public|protected|private)\s+\$\w+\s*;" src/ --include="*.php" | grep -v "= "
grep -rP "var\s+\$" src/ --include="*.php"
```

**怎么修**：给所有属性加上类型声明。动态属性考虑重构为有类型的数组或集合。

---

### CQ-005：错误抑制符（`@`）

**查什么**：用 `@` 抑制错误，如 `@file_get_contents()`、`@unlink()`。

**为什么重要**：`@` 会吞掉所有错误，不止你预期的那一个。它让 bug 隐身，而且速度更慢（PHP 内部要切换 error_reporting 级别）。几乎都有更好的做法。

**怎么检测**：
```bash
grep -rP "@\w+\(" src/ --include="*.php"
grep -rP "@\$\w+" src/ --include="*.php"
```

**怎么修**：换成恰当的错误处理——`try/catch`、检查返回值、或针对特定场景用 `set_error_handler`。

---

### CQ-006：SQL 注入风险

**查什么**：SQL 查询中出现字符串拼接或插值。

**为什么重要**：即使是在提供 query builder 的库中，拼接用户输入的原始 SQL 字符串也是危险的。

**怎么检测**：
```bash
grep -rPi "(SELECT|INSERT|UPDATE|DELETE).*\$" src/ --include="*.php"
grep -rP '".*(SELECT|INSERT|UPDATE|DELETE).*\."' src/ --include="*.php"
```

**怎么修**：用 prepared statement 或者 query builder。绝对不要把用户输入拼进 SQL。

---

### CQ-007：`eval()` 或动态代码执行

**查什么**：`eval()`、`create_function()`、字符串形式的 `assert()`、基于用户输入的变量函数调用。

**为什么重要**：这些都是代码注入的入口，也让静态分析完全失灵。

**怎么检测**：
```bash
grep -rP '\beval\s*\(' src/ --include="*.php"
grep -rP '\bcreate_function\s*\(' src/ --include="*.php"
```

**怎么修**：直接删除。如果确实需要动态行为，用命名可调用对象、策略模式或者白名单。

---

## 🟡 HIGH：现代化改造

### CQ-101：可用构造函数属性提升（8.0+）

**查什么**：构造函数参数被手动赋值给同名属性（`$this->prop = $param`）。

**为什么重要**：构造函数属性提升可以减少约 60% 的样板代码。PHP 8.0+ 支持 `__construct(private string $name)`——一次声明同时完成属性定义、参数声明和赋值。

**怎么检测**：找构造函数中 `$this->prop = $param` 模式的类：
```bash
grep -rP "this->\w+\s*=\s*\$" src/ --include="*.php"
```

**什么时候不要建议**：赋值时包含逻辑（验证、转换）、计算属性、属性可见性和 promotion 能表达的不一致时。

---

### CQ-102：`switch` 可以换成 `match`（8.0+）

**查什么**：用来做值→结果映射的 `switch` 语句。

**为什么重要**：`match` 是表达式（有返回值），使用严格比较，而且不需要 `break`。更不容易出错。

**什么时候建议**：简单的值到值映射的 switch。每个 case 里有复杂逻辑的、或者有 fall-through 的不要建议。

---

### CQ-103：类常量可以换成枚举（8.1+）

**查什么**：在类中定义了一组相关常量，用于表示选项集合（状态、类型、角色）。

**为什么重要**：枚举提供类型安全、可以定义方法、支持 backed value。`Status::Active` 是一个真正的类型——你不能把任意字符串传给期望 Status 类型的参数。

**什么时候建议**：同一个类中有 3 个以上相关常量，且这些常量在代码中作为方法参数或返回值广泛使用时。

---

### CQ-104：缺少 `readonly`（8.1+）

**查什么**：只在 `__construct` 中赋值、之后从未修改的属性。

**为什么重要**：readonly 让不可变性显式化。它既文档化了意图，也防止了意外修改。和构造函数属性提升结合使用：`public readonly string $id`。

**怎么检测**：看构造函数注入的属性。如果它们只在 `__construct` 中被赋值，就该加 `readonly`。

---

### CQ-105：缺少 `#[\Override]`（8.3+）

**查什么**：重写了父类方法但没有加 `#[\Override]` 注解。

**为什么重要**：`#[\Override]` 能捕获笔误和破坏性变更——如果父类方法被重命名或删除，PHP 会直接报 fatal error，而不是静默创建一个新方法。

**怎么检测**：子类中方法名和父类/接口方法名一致的。

---

### CQ-106：可用 nullsafe（8.0+）

**查什么**：多层 null 检查链或三元运算符判断 null。

**为什么重要**：nullsafe 操作符（`?->`）让可选链式调用清晰可读：`$user?->getAddress()?->getCity()` 替代嵌套 null 检查。

**什么时候建议**：返回可空对象的方法后面跟着对该对象的连续方法调用。

---

## 🔵 MEDIUM：结构与性能

### CQ-201：PSR-4 / 命名空间对齐

**查什么**：命名空间与目录路径不一致。

**为什么重要**：命名空间不对齐会导致自动加载失败，IDE 也困惑。命名空间的每一段必须对应目录层级。

**怎么检测**：对比 `composer.json` 的 PSR-4 映射和实际文件位置。`src/Service/UserService.php` 对应 `namespace Vendor\Service;`。

---

### CQ-202：上帝类 / 单一职责违规

**查什么**：类有太多 public 方法（>10-15）、太多构造参数（>5）、或混杂了不相关的职责。

**为什么重要**：上帝类难以测试、难以理解、多种原因都会导致修改。单一职责原则：一个类应该有且只有一个修改的理由。

**怎么检测**：扫关键类。数 public 方法数量和构造函数参数数量。留意类名中有 "And" 的，或者叫 "Manager"/"Handler"/"Processor" 但职责庞杂的。

---

### CQ-203：硬编码依赖

**查什么**：类在内部直接 `new` 实例化依赖，而不是通过构造函数接收。

**为什么重要**：硬编码依赖导致无法单元测试，也不能替换实现。依赖注入让你可以用 mock 测试，换实现不用动类本身。

**怎么检测**：类方法内部（非构造函数、非工厂类）的 `new` 调用。

---

### CQ-204：手动循环可换数组函数

**查什么**：可以用 `array_map`、`array_filter`、`array_reduce`、`array_column` 替代的 `foreach` 循环。

**为什么重要**：原生数组函数是 C 实现的，通常比手动 PHP 循环快 2-5 倍，熟悉之后也更易读。

**什么时候建议**：简单的转换、过滤、聚合循环。

---

### CQ-205：正则可换字符串函数

**查什么**：用 `preg_match`/`preg_replace` 做简单的字符串操作。

**为什么重要**：正则比原生字符串函数慢且难以阅读。字面量匹配优先使用 `str_contains()`、`str_starts_with()`、`str_ends_with()`（PHP 8.0+）或 `str_replace()`。

---

### CQ-206：应使用生成器

**查什么**：返回大数组的方法可以改成 yield 逐项产出。

**为什么重要**：生成器（`yield`）无论数据集多大都使用恒定内存。一个返回 100,000 个元素的数组会耗尽内存；yield 出来几乎不占内存。

**什么时候建议**：构建并返回大数组的方法，尤其是调用方会逐个遍历这些元素时。

---

## 扫描策略

### 不要逐文件全读

对典型的库项目：

1. **先读关键文件**——3-5 个最重要的类（核心服务、主要实体、关键接口）
2. **grep 批量扫描**——`strict_types`、`@` 抑制符、`eval`、缺少类型等
3. **抽样代表性文件**——每个子目录 1-2 个
4. **发现高频问题**——再快速 grep 全仓统计
5. **干净的文件就一笔带过**——不用细看

### 跳过这些

- `vendor/`——永远不看
- `tests/`——对 strict_types 可以放宽（测试框架通常不需要）
- 自动生成的代码（如 `*.generated.php`）——标记但不建议手动修改
- 迁移脚本——跳过

### 节省 Token

- 用 `grep -c` 统计出现次数，不用读文件内容
- 用 `head -50` 瞄一眼文件开头检查 strict_types
- 只在需要理解具体代码模式时才完整读文件
