# Conventional Commits 规范参考

## 基本格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

- **type**: 必填，小写英文，表示提交类型
- **scope**: 可选，小写英文，表示影响范围
- **subject**: 必填，简短描述（本技能使用中文）
- **body**: 可选，详细描述改动内容和原因
- **footer**: 可选，关联 issue、breaking change 等

## Type 类型详解

### feat — 新功能
**含义**：新增一个对用户可见的功能或能力。

**典型场景**：
- 新增 API 端点
- 新增页面或组件
- 新增命令或选项
- 新增数据处理能力

**示例**：
```
feat(user): 添加用户头像上传功能
feat(api): 新增订单批量导出接口
feat(search): 支持全文搜索
```

### fix — 缺陷修复
**含义**：修复一个不期望存在的 bug 或错误行为。

**典型场景**：
- 修复逻辑错误
- 修复边界条件处理
- 修复异常或崩溃
- 修复数据错误

**示例**：
```
fix(order): 修复金额计算精度丢失问题
fix(auth): 修复令牌过期后无限重试的问题
fix(ui): 修复移动端表格横向滚动失效
```

### refactor — 重构
**含义**：不改变外部行为的代码结构调整。

**关键判断**：对外部行为无影响，但代码结构、组织方式发生了变化。

**典型场景**：
- 提取公共函数或模块
- 重命名变量、函数、类（使其更清晰）
- 调整目录结构
- 简化复杂逻辑
- 设计模式引入

**示例**：
```
refactor(payment): 提取支付渠道公共逻辑
refactor(utils): 将日期处理函数迁移到 date-utils 模块
refactor(api): 统一错误处理中间件
```

### docs — 文档
**含义**：仅影响文档的改动。

**典型场景**：
- README 更新
- API 文档更新
- 代码注释（仅注释的改动）
- 使用说明

**示例**：
```
docs(readme): 更新本地开发环境搭建步骤
docs(api): 补充订单接口的错误码说明
```

### style — 代码风格
**含义**：不影响代码逻辑的格式调整。

**注意**：不是 CSS/UI 样式的修改！UI 样式的修改通常属于 feat 或 fix。

**典型场景**：
- 缩进、空格、换行调整
- 缺少分号补充
- 引号风格统一
- import 排序

**示例**：
```
style: 统一使用单引号
style(utils): 修复代码缩进
```

### test — 测试
**含义**：新增或修改测试代码。

**典型场景**：
- 新增单元测试
- 补充测试用例
- 修复测试代码的问题
- 测试框架配置

**示例**：
```
test(order): 补充订单退款的单元测试
test(api): 修复认证中间件的测试用例
```

### chore — 杂项维护
**含义**：不影响生产代码的维护性改动。

**典型场景**：
- 依赖版本更新
- 构建脚本调整
- CI/CD 配置修改
- .gitignore 修改
- 开发工具配置

**示例**：
```
chore(deps): 升级 axios 到 1.6.0
chore(ci): 添加 PR 自动检查流水线
```

### perf — 性能优化
**含义**：以提升性能为主要目的的代码改动。

**典型场景**：
- 算法优化
- 缓存引入
- 懒加载
- 查询优化
- 资源加载优化

**示例**：
```
perf(query): 优化用户列表分页查询性能
perf(image): 引入图片懒加载减少首屏加载时间
```

### ci — 持续集成
**含义**：CI 配置和脚本的改动。

**注意**：和 chore(ci) 的区别——ci 是专门的 type，当改动主体是 CI 系统配置时优先使用。

**示例**：
```
ci: 配置自动部署到测试环境
ci: 添加代码覆盖率检查
```

### build — 构建系统
**含义**：影响构建系统或外部依赖的改动。

**示例**：
```
build(deps): 迁移 webpack 到 vite
build: 调整 TypeScript 编译目标到 ES2022
```

## Scope 命名建议

scope 应当简短、有意义，反映实际影响的模块或组件：

- 按模块：`auth`, `user`, `order`, `payment`, `search`
- 按层级：`api`, `ui`, `db`, `config`
- 按文件：`readme`, `utils`, `middleware`
- 不加 scope 也是可以的（当改动跨多个模块时）

## 正文（Body）写作指南

好的正文回答三个问题：
1. **改了什么** — 具体的改动内容
2. **为什么改** — 改动的原因和背景
3. **有什么影响** — 需要注意的副作用或关联影响

**好的示例**：
```
fix(order): 修复并发下单时库存扣减超卖的问题

将库存扣减操作从乐观锁改为悲观锁，在事务级别加行级锁。
之前在高并发场景下，多个请求同时读到同一库存数，
导致扣减后库存为负数。

关联 Issue: #1234
```

**不好的示例**（只说做了什么，没说为什么）：
```
fix(order): 修改库存更新逻辑

加了锁。改了一下 SQL。
```

## Footer 使用

Footer 用于关联 issue 或标记 breaking change：

```
feat(api): 统一 API 响应格式

BREAKING CHANGE: 所有 API 响应结构从 { data, code, msg } 改为 { data, error }
关联 JIRA: PROJ-456
```

## 常见错误

| 错误 | 正确 | 说明 |
|------|------|------|
| `feat: 修了一个bug` | `fix: 修复xxx问题` | 修 bug 是 fix 不是 feat |
| `fix: 优化了性能` | `perf: 优化xxx性能` | 性能优化用 perf |
| `chore: 重构了代码` | `refactor: 重构xxx` | 重构用 refactor |
| `Feat: xxx` | `feat: xxx` | type 必须小写 |
| `feat(用户): xxx` | `feat(user): xxx` | scope 用英文 |
